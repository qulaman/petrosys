# ТЗ для Claude Code: QuarryOps — система учёта техники карьера (ГСМ / часы / рейсы)

## Контекст проекта

**Новый standalone-проект с нуля.** Это самостоятельный продукт для учёта работы техники и ГСМ на карьерах/стройплощадках, потенциально тиражируемый на других заказчиков. Первый клиент — ТОО «West Arlan Group» (карьер под Актобе).

Стек: **Next.js (App Router, TypeScript) + Supabase (Postgres, Auth, Storage, Realtime) + Tailwind + shadcn/ui**, деплой на Vercel. Инициализируй проект с нуля по этим технологиям.

**Мультитенантность с первого дня.** Таблица `organizations (id, name, created_at)`, колонка `org_id uuid not null` во ВСЕХ доменных таблицах, и каждая RLS-политика включает проверку `org_id = (select org_id from profiles where id = auth.uid())`. Сейчас организация одна, но ретрофит мультитенантности в живую базу недопустимо дорог. Пользователь принадлежит одной организации (profiles.org_id).

## Бизнес-контекст (важно для понимания логики)

Карьер. Работает техника двух типов учёта:
- **По моточасам** — техника на месте: грейдеры, экскаваторы, бульдозеры, катки.
- **По рейсам** — самосвалы (Шахман, Хово), возят грунт с плечом откатки 6+ км.

Водители и техника — **субподрядчики** (сторонние компании), не штатный персонал. Топливо субподрядчикам выдаёт заказчик, объёмы учитываются для взаиморасчётов при закрытии месяца.

Заправка топливом двух типов:
1. **По топливной карте на АЗС** (в основном самосвалы). Карт всего 4, они общие, НЕ выдаются водителям — при заправке присутствует сотрудник заказчика с картой. Он и фиксирует заправку.
2. **С бензовоза на объекте** (грейдеры, экскаваторы и пр.). Раздаёт заправщик — сотрудник заказчика. Бензовоз ведётся как склад на колёсах с балансом.

Подпись водителя на выдачу топлива и на смену — юридически значимое подтверждение (основание для взаиморасчётов). Реализуется как рукописная подпись на canvas (пальцем на телефоне/планшете сотрудника заказчика).

## Роли пользователей

| Роль | Что делает |
|---|---|
| `fueler` (заправщик/держатель карты) | Фиксирует выдачи топлива (оба типа), приход топлива в бензовоз |
| `itr` (ответственный ИТР/диспетчер) | Заполняет табель смены (часы), подтверждает записи своей подписью |
| `checker` (учётчик на разгрузке) | Фиксирует рейсы самосвалов в точке разгрузки |
| `office` (офис/бухгалтер) | Загружает выписки по картам, разбирает расхождения, отчёты, справочники, формирует документы |
| `contractor` (портал субподрядчика) | Read-only доступ к данным СВОИХ договоров: рейсы, часы, топливо, начисления, акты. profiles.contractor_id обязателен для этой роли; RLS всех доменных таблиц для роли contractor фильтрует по цепочке vehicle.contractor_id / contract.contractor_id = profiles.contractor_id. Никаких insert/update |
| `admin` | Всё выше + управление пользователями и справочниками |

Один пользователь может иметь несколько ролей. Auth: Supabase Auth (email или phone), таблица `profiles (id fk auth.users, org_id, full_name, roles text[])`. После логина каждая роль попадает сразу на СВОЙ рабочий экран (см. UX-принципы), а не на общее меню.

    -- equipment (услуги техники): только час
  billing_period text not null default 'monthly' check (billing_period in ('monthly','15days')),

  valid_from date not null,

  is_active boolean default true

-- Прайс-лист договора: цена = f(вид техники, единица измерения), effective-dated
-- Доп. соглашения меняют ставки — это НОВАЯ строка с новым valid_from, старую не трогать
price_list (
  id uuid pk,
  contract_id uuid fk not null,
  vehicle_type text not null,        -- dump_truck, grader, excavator, dozer, roller, water_truck
  unit text not null check (unit in ('trip','hour')),

  vehicle_id uuid fk -> vehicles,    -- null = для всего вида техники; задан = override конкретной машины

--   → максимальный valid_from <= даты записи. Ставка не найдена = запись в закрытии с флагом "нет тарифа".

-- Цена ГСМ для удержания, effective-dated (тоже "как в договоре": с НДС / без НДС по vat_payer)
contract_fuel_prices (

vehicles (

  fuel_card_id uuid fk -> fuel_cards, -- если приход по карте
  receipt_photo_url text,
  created_by uuid fk -> auth.users not null,

-- Замеры фактического остатка бензовоза (инвентаризация)
tanker_measurements (

fuel_issues (

  tanker_id uuid fk -> tankers,          -- обязателен при source_type='tanker'

Также создай представление (view) `tanker_balances`: расчётный остаток каждого бензовоза = сумма refills − сумма issues (source_type='tanker') от последнего замера + значение последнего замера.

### RLS-политики
- `fueler`: insert в fuel_issues, tanker_refills; select справочников.
- `checker`: insert в trip_records; select справочников.
- `itr`: insert/update своих shift_records; select справочников.
- `office`/`admin`: полный доступ на чтение, office — insert card_transactions, update match_status, справочники.
- Все аутентифицированные: select собственных записей.

### Storage
Бакет `signatures` (private) для PNG подписей, бакет `receipts` (private) для фото чеков. Доступ через signed URLs.

## UX/UI-принципы

Система имеет два принципиально разных контекста использования, и дизайн для них разный:

### Полевой контекст (fueler, checker, itr) — телефон, улица, спешка

- **Правило трёх тапов**: любая типовая запись (рейс, заправка) — не более 3 касаний от домашнего экрана. Всё, что можно предзаполнить — предзаполняется: последний источник топлива, закреплённый маршрут, водитель по последним записям, часы = 10.
- **Крупные таргеты**: минимум 56px, ключевые действия — кнопки на треть экрана. Работают в перчатках.
- **Солнце**: высокий контраст, никакой смысловой нагрузки на тонкие оттенки серого. Тёмная тема с автопереключением по времени — ночные смены реальны (22:00–06:00 по договору).
- **Одна рука**: ключевые действия в нижней трети экрана (thumb zone). Навигация — нижний tab bar, не бургер.
- **Ввод без клавиатуры где возможно**: литры — крупный числовой кейпад с пресетами частых значений (150/200/250/300); техника — QR или сетка карточек с гигантскими госномерами; никаких выпадающих списков мельче 48px.
- **Optimistic UI + очередь**: запись мгновенно появляется в локальной ленте со статусом «отправляется», отправка фоном, при ошибке сети — retry-очередь в localStorage с видимым индикатором «N записей не отправлено». Потеря записи в поле недопустима, блокирующих спиннеров нет.
- **Подпись**: полноэкранный canvas, сверху крупно ФИО подписанта («Булгаев, распишитесь»), кнопки Очистить/Готово большие, внизу. 
- **Отмена вместо подтверждения**: для рейса — не диалог «Вы уверены?», а мгновенная запись + 5-минутное окно undo в ленте. Скорость потока важнее.
- **Хоумскрин роли**: fueler открывает приложение → сразу экран выдачи; checker → сразу кнопка рейса. Ноль навигации до основного действия.

### Офисный контекст (office, admin) — desktop, анализ

- Плотные таблицы с сортировкой, фильтрами по периоду/подрядчику/машине, экспорт в Excel из ЛЮБОЙ таблицы.
- Сквозной период-селектор в шапке (сегодня / 7д / 15д / месяц / произвольный) — применяется ко всем разделам.
- **Принцип «нет цифры без расшифровки»**: каждое число на дашборде кликабельно и раскрывается в список записей, из которых оно сложилось, вплоть до конкретной заправки с фото чека и подписью. Это главный механизм доверия к системе.

## Экраны (Next.js App Router)

Всё под общим сегментом, например `/fleet/*`. Мобайл-фёрст обязательно: экраны fueler/checker/itr используются с телефона на улице — крупные кнопки, минимум ввода, работа в ярком солнце (высокий контраст).

### 1. `/fleet/fuel/issue` — выдача топлива (роль fueler)
Один флоу для обоих контуров:
1. Переключатель источника: [Карта №1..4] / [Бензовоз X] (запоминать последний выбор).
2. Выбор техники: сканирование QR (html5-qrcode или аналог) ИЛИ поиск по списку с фильтром по гос. номеру. При выборе техники автоматически подставляется водитель, закреплённый последними записями (можно сменить).
3. Ввод литров (числовая клавиатура), опционально одометр/моточасы.
4. Фото чека (input capture="environment") — для source_type='card' сделать поле обязательным по умолчанию (конфиг).
5. Canvas подписи водителя (библиотека `signature_pad`), кнопка «Очистить». Подпись обязательна.
6. Кнопка «Выдать» → upload подписи и фото в Storage → insert. Показать подтверждение с итогом. Гео-координаты снимать через navigator.geolocation, не блокировать сохранение при отказе.

При source_type='tanker' показывать текущий расчётный остаток бензовоза; предупреждение (не блок), если литров выдаётся больше остатка.

### 2. `/fleet/fuel/tanker` — бензовоз (роль fueler)
- Текущий расчётный остаток.
- «Приход топлива»: литры, цена, источник, карта (если по карте), фото чека.
- «Замер остатка»: ввод фактических литров → система сохраняет рядом расчётный остаток на этот момент и показывает расхождение.
- История операций бензовоза.

### 3. `/fleet/shifts` — табель смены (роль itr)
1. Выбор даты и смены (день/ночь), вида работ.
2. Список активной техники с accounting_type='hours', чекбоксы. Часы по умолчанию 10, инлайн-редактирование.
3. Для каждой отмеченной машины — водитель (автоподстановка по последним записям).
4. Режим подписей: по очереди для каждого водителя открывается canvas на весь экран («Иванов, распишитесь») → следующий. В конце — одна подпись ИТР на всю пачку.
5. Сохранение батчем. Уникальность (vehicle, date, shift) — при конфликте показать существующую запись, предложить редактировать.

### 4. `/fleet/trips` — фиксация рейсов (роль checker)
Максимально быстрый экран, одна рука:
1. Закреплённый маршрут выбирается один раз в начале смены (хранить в localStorage).
2. Крупная кнопка «Сканировать QR» → машина определена → авто-водитель → большая кнопка «Рейс принят». Всё, запись создана.
3. Альтернатива QR — сетка карточек-машин (самосвалы, крупно гос. номера), тап по карточке = рейс с подтверждающим диалогом.
4. Внизу — лента последних рейсов с временем, для самоконтроля. Возможность удалить свою запись в течение 5 минут (soft-delete или window-check в RLS).

### 5. `/fleet/office/transactions` — сверка по картам (роль office)
1. Загрузка выписки: upload CSV/XLSX → парсинг (колонки: дата-время, карта, литры, сумма, АЗС; сделай маппинг колонок настраиваемым при первом импорте, сохраняй пресет маппинга). Дедупликация по unique-констрейнту.
2. Автоматический матчинг: транзакция ↔ fuel_issue по той же карте, |время| ≤ 60 мин, |литры| ≤ 2 л. Однозначная пара → match_status='auto'.
3. Экран разбора: слева несматченные транзакции, справа несматченные issues, ручная привязка (manual) или пометка ignored с комментарием.
4. Виджет: «транзакций без записи в системе: N, литров: X» — главный флаг утечки.

### 6. `/fleet/dashboard` — дашборд (office/admin)

Дашборд — витрина продукта и главный экран директора. Четыре вкладки + центр аномалий. Графики — recharts. Каждая метрика кликабельна (drill-down до записей). Везде сравнение с предыдущим аналогичным периодом (Δ в % рядом с числом).

#### Вкладка «Сегодня» (оперативная, автообновление через Supabase Realtime)
- Строка ключевых чисел: техника на линии X/Y (есть запись смены/рейса/заправки сегодня), рейсов сегодня (+ темп рейсов/час против среднего), часов записано, литров выдано (карта/бензовоз раздельно).
- Живая лента событий: рейс/заправка/смена с временем, машиной, кто записал — realtime-подписка.
- Остатки бензовозов (расчётные) с индикатором «давно не было замера» (> 7 дней).
- Панель «Требует внимания»: сегодняшние аномалии (см. центр аномалий) кратко.
- Мини-карта активности: последние гео-точки записей (рейсы, заправки) на карте — визуально видно, что учёт идёт по всему объекту.

#### Вкладка «Топливо»
- График выдач по дням (stacked bar: card/tanker) за период + линия 7-дневного среднего.
- **Расход к нормативу**: horizontal bar по машинам — фактический л/моточас против fuel_norm_per_hour (линия-риска на баре); превышение — красный. Это договорной инструмент претензий, самый важный график вкладки. Для trips-машин — л/рейс против среднего по типу.
- Сверка карт: donut «сматчено авто / вручную / не сматчено» + сумма «ничейных литров» за период крупно (это прямые деньги).
- Бензовозы: линия расчётного остатка во времени с точками замеров; расхождение замер-расчёт по каждому замеру (bar, красный при > порога).
- Таблица топ-10 потребителей топлива с трендом (sparkline на строку).
- Деньги: стоимость выданного топлива к удержанию по договорам за период.

#### Вкладка «Работа»
- **Тепловая карта активности**: матрица машина × день, цвет = часы или рейсы. Простои и провалы видны за секунду — самый информативный виджет вкладки.
- Рейсы: график по дням; распределение по маршрутам; **гистограмма интервалов между рейсами** по выбранной машине с отметкой медианы — подозрительно короткие интервалы видны как левый хвост.
- Часы: stacked bar по видам работ; выработка часов/день по каждой единице против плановых 10.
- Простои: таймлайн по машинам с цветом вины (подрядчик/заказчик), сумма неоплачиваемых часов и компенсируемых (правило 18:00).
- Выработка самосвалов: рейсов/смену по машинам, медиана по парку как бенчмарк.

#### Вкладка «Подрядчики и деньги»
- Начислено нарастающим итогом по каждому договору за текущий расчётный период + **прогноз суммы АВР** (линейная экстраполяция по темпу) — директор видит будущий платёж до его наступления.
- Структура: начислено / удержано ГСМ / штрафы — waterfall или stacked bar по договорам.
- **Эффективная стоимость**: полная стоимость рейса (ставка минус удержанный ГСМ, плюс наш ГСМ если не удерживается) и часа по каждому подрядчику — сравнение подрядчиков между собой на одном bar chart. Основа для переговоров по ставкам.
- Тенге/м³ перевезённого грунта (рейс = min 19 м³ по договору, объём — настройка маршрута) — главная метрика себестоимости для директора.
- Доля каждого подрядчика в объёме работ (рейсы/часы) — donut.

#### Центр аномалий `/fleet/dashboard/anomalies`
Единая лента флагов с воркфлоу разбора. Таблица `anomalies (id, org_id, type, severity, entity_refs jsonb, detected_at, status check in ('new','reviewed','confirmed','dismissed'), resolution_note, reviewed_by)`. Детект — pg_cron-джоб ежедневно + on-demand пересчёт. Типы:
1. `fuel_no_work` — топливо выдано, работы за день нет
2. `work_no_fuel` — техника работает N дней без заправок (N по типу техники)
3. `over_norm` — расход выше норматива договора (severity high — договорное нарушение)
4. `short_trip_interval` — интервал между рейсами < 50% медианы машины
5. `driver_double_shift` — один водитель в день+ночь одной даты
6. `hours_over_11` — превышение лимита рабочего времени
7. `unapproved_unit` — запись на технику/водителя вне периода допуска
8. `tanker_gap` — расхождение замера и расчёта бензовоза > порога
9. `unmatched_txn` — транзакция по карте без записи в системе > 24ч
10. `continuous_driving` — рейсы без перерыва ≥ 15 мин на протяжении > 4ч

Каждый флаг: статус, комментарий разбора, ссылки на записи-основания. Виджет «новых аномалий: N» — в шапке всех офисных экранов. Подтверждённые аномалии типов 3, 8 могут конвертироваться в penalty одной кнопкой (предзаполненное основание).

### 7. `/fleet/office/settlement` — закрытие периода по договору (office)
- Выбор **договора** (не подрядчика — у подрядчика может быть несколько договоров с разными условиями) и расчётного периода. Период определяется contract.billing_period: месяц или 15 дней.
- Начислено: каждая запись (рейс/смена) × ставка из price_list. Разрешение ставки: точное совпадение (contract, unit, vehicle_id) → по виду (contract, unit, vehicle_type), с максимальным valid_from ≤ даты записи. КРИТИЧНО: доп. соглашения меняют ставки посреди периода — записи до/после даты допника считаются по разным строкам прайса. Записи без найденной ставки — в отдельный блок «нет тарифа», в итог не включать.
- Удержано ГСМ: литры × цена из contract_fuel_prices (effective-dated по дате выдачи). Нет записей цены — топливо показывается справочно без удержания.
- Удержано штрафов: непогашенные penalties по договору, с пометкой периода погашения.
- НДС: вычисляется по contractors.vat_payer. Плательщик — из всех сумм выделяется НДС отдельной строкой акта (сумма × 16/116, цены хранятся с НДС), в оплате участвует ЭСФ. Неплательщик — строк НДС в акте нет.
- Итог: начислено − ГСМ − штрафы + детализация. Экспорт в Excel (библиотека, принятая в проекте, иначе exceljs) — акт сверки с построчной детализацией: каждая выдача топлива с датой и подписью-ссылкой, каждая смена, каждый рейс. Формат акта — основа для АВР по договору.

### 8. `/fleet/admin/*` — справочники (admin/office)
CRUD: contractors, vehicles (+генерация и печать QR-наклеек — страница с QR-кодами для печати, библиотека qrcode), drivers, fuel_cards, tankers, routes, work_types, пользователи и роли.

### 9. `/fleet/office/documents` — формирование документов (office)
- Карточка контрагента с полными реквизитами (БИН/ИИН, адрес, банк, ИИК, БИК, руководитель, vat_payer).
- Мастер создания договора: шаблон (transportation/equipment) → контрагент → условия (ставки, цена ГСМ, период, срок). Введённые условия ОДНОВРЕМЕННО создают contract + price_list + contract_fuel_prices и подставляются в docx-шаблон. Двойного ввода нет — это ключевое требование.
- Кнопки генерации по типам из generated_documents.doc_type. Автоматические триггеры: изменение прайса → предложение сформировать доп. соглашение; изменение состава техники/водителей → новая редакция Приложения №2 (номер редакции автоинкремент); подтверждённый сигнал over_norm → претензия с предзаполненным расчётом; закрытие периода → пакет АВР + акт сверки + реестр рейсов + ведомость ГСМ.
- История документов в карточке договора, автонумерация по типу.
- Генерация: docx-шаблоны с плейсхолдерами (docxtemplater), хранение в Storage-бакете documents (private, signed URLs).

### 10. `/portal/*` — портал субподрядчика (роль contractor)
Отдельный layout, только чтение, только данные своего contractor_id:
- Главная: начислено нарастающим итогом по каждому своему договору за текущий период + прогноз АВР.
- Мои рейсы (время каждой ходки), мои смены, выданное топливо (с подписями своих водителей и фото чеков).
- Мои акты и документы — выгрузка из generated_documents своих договоров.
- Никакой аналитики компании, чужих контрагентов, сигналов, цен других договоров. RLS — жёстко по contractor_id, проверить изоляцию тестами.

## Технические требования

- **PWA**: manifest + иконки, полевые сотрудники ставят на главный экран. Полноценный offline-first НЕ делать в первой версии, но: retry-очередь в localStorage для неотправленных записей выдач/рейсов/смен с видимым счётчиком — потеря записи в поле недопустима.
- Подписи: `signature_pad`, экспорт в PNG (toDataURL), upload в Storage до insert записи; при фейле upload — не терять данные формы.
- Realtime: подписка на новые записи для живой ленты дашборда «Сегодня» (Supabase Realtime, канал по org_id).
- Аномалии: pg_cron ежедневный джоб + RPC для пересчёта on-demand.
- Все даты хранить в timestamptz, отображать в Asia/Aqtobe (UTC+5).
- Валидация на сервере (route handlers / server actions) дублирует клиентскую.
- Язык интерфейса: русский. Строки — простой словарь-объект для будущего казахского, i18n-фреймворк не тянуть.
- Тёмная тема (для ночных смен) с автопереключением, переключатель в настройках.
- Маршрут (routes) получает поле `volume_m3 numeric(6,2)` — объём кузова/рейса для метрики тенге/м³ (из договора: min 19 м³/рейс).

## Договорные валидации (реализовать как предупреждения, не жёсткие блоки, если не указано иное)

Эти правила взяты напрямую из договоров с субподрядчиками:

1. **Допуск**: запись рейса/смены/заправки на технику или водителя вне периода approved_from/approved_to — предупреждение «не в актуальном Приложении №2». Техника/водители вне списка к работе не допускаются, оплата таких записей — риск заказчика.
2. **Смены**: один и тот же водитель в дневную (06:00–22:00) и ночную (22:00–06:00) смену одной даты — флаг (договор требует полную замену водителя).
3. **Часы**: hours > 11 в записи табеля — предупреждение (лимит рабочего времени по договору).
4. **Расход ГСМ**: фактический л/час по машине за период > fuel_norm_per_hour — алерт на дашборде. Это договорное нарушение (расход сверх норматива без письменного согласования запрещён), основание для претензии.
5. **Простой заказчика**: downtime с fault_side='client' и notified_at позже 18:00 предыдущего дня — пометка «подлежит компенсации 20% дневной ставки».
6. **Интервалы рейсов**: непрерывное вождение по договору ≤ 4 часов — рейсы одного водителя без перерыва ≥ 15 мин на протяжении > 4 часов — флаг.

## Открытые вопросы (спроси у меня, если критично, иначе делай дефолт)

- Формат выписки топливного оператора неизвестен — маппинг колонок настраиваемый.
- Обязательность подписи водителя на рейсе: договор требует подпись водителя на ТТН, но подпись на каждой ходке замедлит поток на разгрузке. Дефолт: настраиваемый флаг на уровне маршрута, начать без подписи (авторизация учётчика + timestamp + гео), подпись добавить если бухгалтерия/подрядчик потребуют.
- Простои (downtime_records) и штрафы (penalties): таблицы создать в Фазе 1, интерфейсы — в Фазе 5 вместе с закрытием периода.
